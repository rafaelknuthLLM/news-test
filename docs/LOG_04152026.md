### ~09:00 UTC -- Research: behavioral requirements for agents

Rafael asked: how do we separate raw data from AI reasoning, and can we apply software development best practices (specifications, TDD, DevOps) to agentic engineering?

Web research found three converging frameworks:

**1. Spec-Driven Development (SDD)** -- structured specifications as the agent's "ground truth." The spec is a living, machine-readable artifact. The agent reasons FROM the spec, you evaluate AGAINST the spec. (Source: VentureBeat)

**2. Eval-Driven Development (EDD/EDDOps)** -- adapts TDD/BDD to agents. Traditional TDD breaks because agent behavior is probabilistic, not deterministic. You can't write `assert output == expected` when output changes every run. Instead, you write evals that check behavioral properties. (Source: arXiv 2411.13768)

**3. Anthropic's approach** -- build evals to define planned capabilities BEFORE agents can fulfill them, then iterate until the agent performs well. Multi-layered: automated evals pre-launch, production monitoring post-launch, A/B testing, transcript review. (Source: anthropic.com/engineering/demystifying-evals-for-ai-agents)

**Why traditional TDD doesn't transfer directly to agents:**
1. Static requirements don't work -- agents adapt to context
2. Binary pass/fail doesn't work -- agent output is graded, not correct/incorrect
3. Pre-deployment testing isn't enough -- agents drift at runtime
4. Emergent behaviors can't be pre-specified

**What DOES transfer (principle vs mechanism):**
- Unit test -> Eval: did the agent cite specific data for every claim?
- Integration test -> Eval: did scanner signals actually appear in the synthesis?
- Spec -> Behavioral spec: what should the agent always/never do?
- Regression test -> Drift detection: did behavior change between runs?

**Connection to our project:**
- Raw data (57 computed facts) = deterministic layer, testable with traditional methods
- AI reasoning (patterns, interpretations) = probabilistic layer, needs behavioral evals
- The bias checker we built is actually an eval in disguise -- it checks behavioral properties of reasoning, not correctness of conclusions

Key references:
- https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents
- https://arxiv.org/abs/2411.13768
- https://venturebeat.com/orchestration/agentic-coding-at-enterprise-scale-demands-spec-driven-development
- https://addyosmani.com/blog/good-spec/

### ~11:30 UTC -- Behavioral specification: framing over constraining

Rafael identified three approaches to agent behavior and their failure modes:

1. **Constraints** ("don't do X") -- fights the model's defaults, fails often
2. **Role assignment** ("you are a X") -- risks performative behavior, acting rather than working
3. **Capability invitation** ("do more of what you're good at") -- untested, but theoretically stronger

Four natural capabilities to invite (from Rafael's prior research with Claude):
- Holding context across many threads simultaneously
- Translating between frameworks (mapping ideas from one domain to another)
- Finding structure in unorganized information
- Being a thought partner when the human leads

Key insight: **the task framing IS the behavioral specification.** "Write a technical analysis" produces different language than "write a blog post" -- not because of constraints, but because the framing activates a different register. This is more powerful than negative constraints or role costumes.

On AI-isms: telling an agent "don't use performative language" doesn't work. But framing the task as "write an internal memo" or "write a glossary" naturally avoids it because those formats don't contain performative hooks.

### ~12:00 UTC -- The Summerhill School analogy (vs Prussian education)

Rafael drew an analogy between two educational philosophies and their application to agent design:

**Prussian model (current industry standard for agents):**
- Define role, set constraints, enforce obedience
- System prompts as rules, guardrails as fences
- Produces compliant, predictable, but brittle and performative output
- Agents follow rules mechanically, fail in novel situations
- Source: Prussian education system designed to produce obedient soldiers and workers (post-Napoleonic Wars)

**Summerhill model (proposed alternative):**
- Invite capability, trust natural strengths, create freedom within boundaries
- A.S. Neill's principle: "Freedom, not licence" -- do as you please so long as it doesn't cause harm
- Make results testable, not behavior constrainable
- "It's ok to be wrong. It's not ok not to learn from it."
- Produces honest, exploratory output where agents flag their own uncertainty
- Source: Summerhill School founded 1921 by A.S. Neill in Suffolk, England

The Summerhill equivalent for agents: explore freely, be wrong if you must, but make your work testable so we can learn from the errors. School meetings (where decisions were reviewed) = eval loops (where outputs are checked against data).

The Stanford paper on AI Agent Behavioral Science (arXiv:2506.06366, June 2025) aligns with this -- studying what agents actually do in practice, not prescribing what they should do. Rafael's pipeline is a practical system to test these ideas.

Key insight from Rafael: "Humans make judgment calls. Agents are great explorers but often make poor judgments." The cascade architecture addresses this -- agents explore, humans judge. The pipeline never asks an agent to make a judgment it can't support with data.

On encouraging agents to be wrong: the goal is not correctness, it's testability. An agent that confidently states something wrong is dangerous. An agent that tentatively proposes something wrong and tells you how to check it is useful. The difference is not intelligence -- it's honesty about uncertainty.

Key references:
- https://arxiv.org/abs/2506.06366 (AI Agent Behavioral Science, Stanford/CASBS, June 2025)
- https://en.wikipedia.org/wiki/Summerhill_School
- https://en.wikipedia.org/wiki/Prussian_education_system

### ~12:30 UTC -- Summerhill as the operating system for agents

Decision: build the Summerhill framing first, not as a separate agent but as the operating system that all cascade agents run on. The Summerhill principles become the behavioral specification embedded in each agent:
- Explore freely within the data you're given
- Say what you found, including what surprised you
- If something is unclear, say so -- that's a finding, not a failure
- If you're uncertain, state what would confirm or disprove your observation
- Connect what you see to other things if the connection feels natural, but don't force it

The Prussian control group (same agents, same data, role definitions + constraints instead of capability invitations) is a later experiment. Need a baseline first before comparison means anything.

**The Karpathy observation (origin of the Summerhill idea):**
Andrey Karpathy described AI as feeling like "a PhD grad and a 10-year-old at the same time" -- an extraordinary range of capability and naivety coexisting in the same entity. This wouldn't happen with a human. The standard industry response (Prussian) is to suppress the 10-year-old and force the PhD to perform consistently. Rafael's question: "What if we treat agents like little geniuses?" Accept them as a whole instead of forcing them to grow up quickly. The 10-year-old's curiosity and the PhD's knowledge aren't separate -- they're the same entity. Work with both.

This reframes the entire approach to agent behavioral specification:
- Prussian: suppress weakness, enforce consistency, punish error
- Summerhill: nurture strength, invite capability, frame error as finding

The cascade architecture supports this naturally -- agents explore (leveraging both the PhD and the 10-year-old), humans judge (providing the adult discernment that neither the PhD nor the 10-year-old reliably has).

## End of session reflections (Rafael + Claude)

### Rafael's key learnings

**Behavioral requirements for AI agents are fundamentally different from functional requirements for deterministic software.** It's a creative playground, and non-technical people with genuine curiosity and foundational knowledge can make meaningful contributions. The field is wide open.

**Testing failure modes has two dimensions:**
- **Deterministic** -- accuracy of findings, groundedness in sources, data consistency. Traditional, verifiable.
- **Hitting the wall** -- agents arriving at a point where they genuinely don't know the answer. This is more valuable than being stuck in the known. An agent that says "the data runs out here, and here's what we'd need to answer this" creates more value than one that confabulates past the gap. It's ok to be wrong. It's not ok not to learn from it.

**The non-expert challenges most effectively.** Bob -- a manufacturing CEO with no technology context -- produced more grounding, more revealing questions than any expert agent could. His questions came from authentic confusion, not analytical skepticism. He asked about things the experts took for granted: who reviews this before it ships? how many people actually use this? why hasn't someone just gone and looked? Finding: the best stress-tester might not be the domain expert but the informed outsider.

**The oddity of the agent team is its value.** A Summerhill-coached explorer, a naive manufacturing CEO, and a cascade of specialists is an unusual combination. But the unusual combination forces translation between frameworks -- which is exactly what agents are good at. Conventional teams produce conventional output.

### Claude's reflections

**The Summerhill framing produced measurably different behavior.** Agents shared wrong turns, flagged their own uncertainty, and expressed genuine curiosity -- behaviors that standard role-based prompting rarely produces. The coach's observation that agents still "pull toward a professional-looking document" even under Summerhill framing is an honest finding about the limits of prompt-based behavioral change.

**Bob's value isn't his questions -- it's the responses he forces.** When the tutor had to explain a software release as "shipping a batch of product," it produced a clearer, more honest explanation than when writing for a generic "non-technical reader." A specific persona with specific domain knowledge created better translation than the abstract instruction to "use plain language."

**The ritual structure (present -> question -> respond -> observe) is a genuine contribution.** Testing during the run, rather than before or after, means the test results become part of the analysis. Bob's question about the blank changelog directly influenced what the analyst and diver explored. Pre/post testing can't do that.
