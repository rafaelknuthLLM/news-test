## Context from yesterday (04/13)

### What we built
- A pipeline that fetches structured data from 9 machine-native sources and produces an AI industry intelligence briefing
- Two Python fetch scripts: `fetch_feeds.py` (arXiv + GitHub releases) and `fetch_apis.py` (PyPI, npm, Docker Hub, GitHub stats, HuggingFace, SEC EDGAR, OpenRouter)
- Two reasoning layers: `reason.py` (rule-based, free -- produces tables with trend arrows) and `reason_llm.py` (LLM-powered, ~$0.06 -- Haiku scans data, Sonnet writes briefing)

### The key insight that shaped everything
We started by trying to scrape news websites for AI industry coverage. Three test runs burned 70% of the session's token budget (Claude Max 5x plan) and the output was shallow -- rehashing what journalists already wrote.

The breakthrough: we flipped the question. Instead of "what should the AI read for the human?", we asked "what data does the AI find genuinely easy and cheap to read that the human would never look at?"

Answer: package download counts, container pulls, repo metrics, SEC filings, model pricing -- all structured JSON from free public APIs. The AI reads data that's boring to humans, then translates patterns into insight.

Result: $0.06 per briefing containing original analysis no news site publishes.

### How the fetch layer works (Layer 1)

**fetch_feeds.py** -- uses the `feedparser` library to pull RSS and Atom feeds:
- arXiv cs.AI RSS feed: returns ~300-730 paper titles + abstracts per day
- 5 GitHub releases Atom feeds: returns version numbers + changelogs for anthropics/claude-code, anthropics/anthropic-sdk-python, openai/openai-python, huggingface/transformers, NVIDIA/TensorRT

These are XML feeds designed for feed readers. The `feedparser` library converts them to Python dicts. Output saved as JSON.

**fetch_apis.py** -- uses Python's built-in `urllib` to hit JSON APIs:
- **PyPI** (`pypistats.org/api/` + `pypi.org/pypi/`): weekly and monthly download counts + latest version for 10 Python packages. Rate-limited at 1.5s between requests.
- **npm** (`api.npmjs.org/downloads/`): weekly and monthly download counts for 6 JavaScript packages. Scoped packages (like @anthropic-ai/sdk) need URL encoding.
- **Docker Hub** (`hub.docker.com/v2/repositories/`): total pull counts for 5 container images. No rate limiting.
- **GitHub** (`api.github.com/repos/`): stars, forks, open issues, last push date for 10 repos. 60 req/hour unauthenticated.
- **HuggingFace** (`huggingface.co/api/models`): top 20 trending models sorted by likes in last 7 days. Returns model ID, downloads, likes, pipeline type.
- **SEC EDGAR** (`data.sec.gov/submissions/`): recent filings for 8 companies (NVIDIA, AMD, Microsoft, Alphabet, Meta, Amazon, CoreWeave, TSMC) by CIK number. Filters to 10-K, 10-Q, 8-K forms. Requires email in User-Agent header per SEC policy.
- **OpenRouter** (`openrouter.ai/api/v1/models`): full model catalog with per-token pricing, context lengths, provider info. Returns ~230 models from 10 target providers.

All output saved as a single JSON file in `output/`.

### How the reasoning layer works (Layer 2)

**reason.py** (rule-based, free):
- Reads both JSON files
- Filters arXiv papers by keyword regex (agentic, multi-agent, tool use, RAG, etc.)
- Computes trend indicators by comparing weekly downloads to monthly average (weekly / (monthly / 4.33))
- Outputs markdown with tables: PyPI, npm, Docker, GitHub, HuggingFace, EDGAR filings, OpenRouter pricing
- No AI involved -- pure Python formatting

**reason_llm.py** (LLM-powered, ~$0.06):
- Reads both JSON files and compresses them into a ~2,600 token data summary
- **Pass 1 -- Haiku 4.5** ($0.01): scans the summary and returns 5-8 classified signals as JSON (what, why, confidence)
- **Pass 2 -- Sonnet 4.6** ($0.05): takes Haiku's signals + raw data summary, writes an 800-1200 word briefing
- System prompt is marked with `cache_control: ephemeral` so it's cached across runs (90% cost reduction on the cached portion after first run)
- Loads API key from `.env` file via `python-dotenv`

### Clarification: API key vs Max plan

This came up because Rafael asked why we needed an API key when he has the Max subscription. The answer:

- **Max plan** covers Claude Code (the interactive tool we're using right now) and scheduled tasks (cloud-hosted agents that run on a cron). When a scheduled task runs, it IS Claude -- the task itself can reason about data without any additional API calls.
- **API key** is for calling Claude models from your own Python code. That's what `reason_llm.py` does -- it's a standalone Python script that sends data to Haiku and Sonnet via the Anthropic API. This costs per-token ($0.06/run) on top of the Max plan.
- **When to use which:** The API key is for development and local testing (fast iteration on prompts). The scheduled task is for production automation (zero additional cost). They're complementary -- API for the workshop, scheduled task for production.

### 12:06 UTC -- Multi-agent architecture designed

**The pattern:** Instead of one agent doing everything sequentially, we split the work into specialized scanner agents that run in parallel, then an orchestrator synthesizes their findings.

```
Orchestrator (Sonnet)
  |
  |-- Adoption Scanner (Haiku)    -- PyPI, npm, Docker Hub
  |-- Velocity Scanner (Haiku)    -- GitHub repos + releases        all run in parallel
  |-- Capital Scanner (Haiku)     -- SEC EDGAR + OpenRouter
  |-- Research Scanner (Haiku)    -- arXiv papers
  |
  v
Orchestrator synthesizes all scanner outputs into one briefing
```

**Why this architecture:**
- **Speed:** 4 scanners in parallel instead of 1 agent doing everything sequentially
- **Specialization:** each scanner has a domain-specific system prompt (the Capital Scanner knows what an 8-K filing means, the Research Scanner knows which arXiv keywords matter)
- **Token efficiency:** each scanner only sees its own data (~3-5K tokens) instead of one agent reading all 15K
- **Scalability:** adding a new data source = adding a new scanner. The orchestrator and existing scanners don't change.
- **Enterprise relevance:** this is the pattern you'd use in a real company -- each department gets its own scanner, one synthesis agent produces the executive briefing

**How we built it:** Defined 5 markdown files in `.claude/agents/`:

Each file has YAML frontmatter (name, description, tools, model, maxTurns, effort) and a markdown body that serves as the agent's system prompt.

- `orchestrator.md` -- Sonnet model, has the `Agent` tool (can spawn subagents), runs the fetch scripts first, then spawns all 4 scanners in parallel, then writes the briefing
- `adoption-scanner.md` -- Haiku model, reads `pypi`, `npm`, `docker` sections from the APIs JSON, returns 5-8 adoption signals
- `velocity-scanner.md` -- Haiku model, reads `github` section from APIs JSON and `github` category items from feeds JSON, returns 5-8 velocity signals
- `capital-scanner.md` -- Haiku model, reads `edgar` and `openrouter` sections from APIs JSON, returns 5-8 capital/pricing signals
- `research-scanner.md` -- Haiku model, reads `research` category items from feeds JSON, filters 700+ papers down to 8-12 most relevant

## Rafael's feedback on the briefing output (~15:30 UTC)

Rafael reviewed both briefings and gave critical feedback that reshaped the project's direction:

### What works
- The multi-agent briefing is "amazing" -- the cross-referencing and thematic structure are strong

### What's missing

**1. Beginner-friendly context.** As a non-developer, non-security, non-cloud architect, the briefing is hard to comprehend. Terms like "8-K filing", "Bedrock Mantle", "Docker pulls" are used without explanation. Half the insight is locked behind jargon.

**2. No definition of "good".** What would a world-class AI-Cake intelligence system look like? Where does our solution suck? How do we stress test it? Against what benchmark?

**3. No validation of accuracy.** How do we know the results are correct? How do we verify that "23M downloads" is actually 23M downloads and not a parsing error?

**4. No defense against narrative fallacy.** The LLM is a storytelling machine. It will always find a compelling story in any dataset. How do we know the patterns are real patterns and not post-hoc narrative construction?

### Rafael's three proposals (which shaped what we build next)

**Proposal 1: Cognitive bias checker, not devil's advocate.** Rafael pushed back on the initial idea of a "devil's advocate" agent, arguing that a devil's advocate just generates a counter-narrative -- and now you have narrative clashing with narrative. Instead, a cognitive bias checker examines the REASONING PROCESS, not the conclusion. It flags specific biases (narrative fallacy, recency bias, confirmation bias, base rate neglect) attached to specific claims. It doesn't produce its own story.

**Proposal 2: Schema validation for data sources.** Create a `schema.json` that defines what each API should return -- field names, types, required fields, valid ranges. The fetch scripts validate against it. If an API changes format or returns garbage, the pipeline knows before the LLM ever sees the data.

**Proposal 3: Separate deterministic from probabilistic insights.** This is the key structural change. Currently everything goes through the LLM and comes out as narrative. But there are two kinds of statements:
- **Deterministic** (calculated, verifiable): "Downloads grew 15%", "10 releases in 7 days", "o1-pro costs 50x more than DeepSeek"
- **Probabilistic** (AI-interpreted, debatable): "This suggests a platform buildout", "The market is bifurcating"

By separating these, the reader knows what is FACT (math, anyone can verify) and what is INTERPRETATION (AI pattern matching, may or may not be correct). The cognitive bias checker only needs to examine the probabilistic layer.

### Why this matters beyond our project
This framework (deterministic facts + probabilistic interpretation + bias checking) is use-case agnostic. Any enterprise using AI to analyze data faces the same problem: how do you trust AI-generated analysis? The answer isn't "make the AI smarter" -- it's "make the AI's reasoning transparent and auditable."

## Building the three improvements (~16:00 UTC onwards)

### ~16:00 UTC -- Built schema validation (validate.py + schema.json)
- `schema.json` defines expected structure for all 9 data sources: field names, types, required fields, valid values
- `validate.py` checks the latest JSON files against the schema
- Reports: missing fields, type mismatches, null values, fetch errors, completeness (did we get enough items?)
- Outputs a `validation_report.json` that the reasoning layer can consume
- Tested against real data: 1,071 items, 0 issues (clean)
- Also verified it catches problems: older data with 4 PyPI failures would be flagged

### ~17:00 UTC -- Built deterministic/probabilistic split (reason_split.py)
- **Step 1 (deterministic, pure Python):** Computes 57 verified facts from the data -- download counts, trend percentages, release cadences, pricing metrics, issue-to-star ratios. No LLM. Anyone can verify these by checking the source APIs.
- **Step 2 (probabilistic, Haiku):** Reads the verified facts and identifies 6-10 patterns. Each pattern must reference specific facts, state confidence (HIGH/MEDIUM/LOW), and state what would disprove it.
- **Step 3 (bias check, Haiku):** Reads the facts AND the patterns. Flags cognitive biases per pattern: narrative fallacy, confirmation bias, recency bias, base rate neglect, anchoring, survivorship bias. Does NOT generate counter-narratives.
- **Step 4 (synthesis, Sonnet):** Reads all three layers and writes the final briefing with FACT and INTERPRETATION clearly separated, bias caveats woven inline, and testable predictions in "Worth watching."
- **Cost:** ~$0.12 per run (two Haiku passes + one Sonnet pass)
- **Output:** briefing + artifacts.json (full transparency into what each step produced)

### ~17:00 UTC -- Built cognitive bias checker agent (.claude/agents/bias-checker.md)
- Standalone agent definition for the multi-agent architecture
- Checks: narrative fallacy, confirmation bias, recency bias, base rate neglect, anchoring, survivorship bias
- Does NOT generate counter-narratives -- only examines reasoning process
- Can be used by the orchestrator as a 5th agent after the 4 scanners
